# Command Line Interface

The Yasumu CLI composes the shared headless workspace and execution
services with the Node.js worker runtime. It uses the same YSL
schemas, request builder, hook lifecycle, tests, interpolation, and
structured results as the desktop app.

<Callout type="info" title="Runtime requirement">
  The headless CLI requires Node.js 24 or newer.
</Callout>

## Workspace discovery

Commands load `$cwd/yasumu` by default. `--workspace` accepts either a
project directory containing `yasumu/` or the `yasumu` directory
itself:

```bash
yasumu validate
yasumu validate --workspace /path/to/project
yasumu validate --workspace /path/to/project/yasumu
```

`--path` remains as a deprecated alias.

The loader discovers every supported `.ysl` file in deterministic
order. It validates the complete workspace, reports file and source
ranges, rejects duplicate IDs and invalid references, and never
silently skips invalid files.

## Commands

```bash
yasumu validate
yasumu list [--kind rest|graphql|sse]
yasumu info
yasumu run <entity-name-or-id>
yasumu test [entity-name-or-id]
yasumu sse list
yasumu sse run <entity-name-or-id> --max-events 10
```

`yasumu test` without an entity runs all REST, GraphQL, and SSE
entities in deterministic order. The previous `yasumu rest list`,
`yasumu rest run <entity>`, and `yasumu rest run --all` forms remain
available as aliases.

### Validate and inspect

```bash
yasumu validate --json
yasumu list --kind graphql
yasumu info
```

### Run an entity

An entity can be selected by stable ID or exact name:

```bash
yasumu run "Create user" --environment Staging
yasumu run get-current-user --timeout 15000 --verbose
```

REST, GraphQL, and SSE use one execution command and one result shape.
SSE results additionally contain the collected event array.

### Connect to an SSE stream

```bash
yasumu sse run deployment-events --max-events 5
yasumu sse run local-stream --max-events 2 --json
```

The CLI hosts the built-in echo server on demand and resolves
`echo.yasumu.local` to it. This keeps checked-in local stream fixtures
portable between the desktop, CLI, and CI. `--max-events` defaults to
10 for the `sse run` command.

### Run tests

```bash
yasumu test get-current-user
yasumu test --environment CI --dotenv .env.ci --json
```

A test command fails when execution fails or an assertion fails.
Scripts still run through `onRequest`, transport or mock,
`onResponse`, then `onTest`.

## Execution options

| Option                           | Description                             |
| :------------------------------- | :-------------------------------------- |
| `-w, --workspace <path>`         | Project or `yasumu` directory           |
| `-e, --environment <name-or-id>` | Active environment                      |
| `--dotenv <path>`                | Explicit dotenv file to load            |
| `--variable KEY=VALUE`           | Typed variable override; repeatable     |
| `--secret KEY=VALUE`             | Secret override; repeatable             |
| `--timeout <milliseconds>`       | Request and script timeout              |
| `--max-events <count>`           | Stop SSE after accepted event count     |
| `--json`                         | Emit one machine-readable JSON document |
| `-v, --verbose`                  | Include bodies in human output          |

JSON literals supplied with `--variable` preserve their type:

```bash
yasumu run create-user \
  --variable RETRIES=3 \
  --variable ENABLE_CACHE=true \
  --variable 'PROFILE={"name":"Ada"}'
```

## Environment and secret precedence

For a CLI invocation, effective values use this deterministic
precedence, from highest to lowest:

```d
CLI --variable / --secret
  → process environment
  → explicit --dotenv values
  → selected workspace environment
  → workspace defaults
```

The CLI never searches for or loads a `.env` file automatically.
Relative `--dotenv` paths are resolved from the current working
directory, while absolute paths are accepted unchanged. The file is
parsed by the shared headless dotenv parser.

Process variables can use `YASUMU_VAR_<NAME>`. Defined secrets can use
`YASUMU_ENV_<NAME>`. A plain process name also overrides a matching
variable or secret already defined by the selected workspace
environment:

```bash
YASUMU_VAR_API_URL=https://api.example.com \
YASUMU_ENV_API_TOKEN="$API_TOKEN" \
yasumu test --environment CI
```

Secrets are supplied explicitly in headless mode and are redacted
centrally from logs, diagnostics, request and response snapshots,
errors, tests, and execution history.

## Files and multipart requests

YSL `workspace-path` references are resolved relative to the loaded
workspace. The CLI rejects traversal and symlink escapes outside that
root. Binary and multipart requests retain file names, MIME types,
repeated fields, and empty files without embedding large binary bodies
in result snapshots.

Scripts can also open a workspace file through
`ctx.workspace.files.open('fixtures/avatar.png')`.

## Scripts and virtual modules

CLI scripts run in isolated Node.js worker threads through
`@yasumu/runtime-node`. They receive the same `RequestHookContext`,
`ResponseHookContext`, and `TestHookContext` used by the desktop
runtime. Standard Web `Request` and `Response`, mock responses, nested
REST/GraphQL/SSE execution, email host calls when configured, file
host calls, logs, tests, and the generated `yasumu:` virtual modules
are supported.

See [Scripting](/docs/scripting) for the exact API.

## Cancellation

Pressing `Ctrl+C` aborts the active request, hook, nested executions,
email waits, file reads, and worker session. The CLI emits a
structured cancellation result and exits with `130`.

## Exit codes

| Code  | Meaning                                                           |
| :---- | :---------------------------------------------------------------- |
| `0`   | Validation or execution succeeded                                 |
| `1`   | Execution, HTTP status policy, or a test failed                   |
| `2`   | Invalid workspace, entity/environment selection, or command input |
| `130` | Cancelled by `SIGINT`                                             |

## CI example

```yaml
- name: Validate Yasumu workspace
  run: pnpm exec yasumu validate --json

- name: Run API tests
  env:
    YASUMU_ENV_API_TOKEN: ${{ secrets.API_TOKEN }}
  run: pnpm exec yasumu test --environment CI --json
```

Machine output is a single serializable document containing request
and response snapshots, tests, logs, diagnostics, nested execution
summaries, timing, status, and a structured error when applicable.
