# Testing

Tests are declared in `onTest(ctx)` and run after `onResponse` only
when an entity is executed in test mode. The desktop app and CLI
return the same structured test results.

```ts
export async function onTest(ctx: TestHookContext) {
  const body = await ctx.res.clone().json();

  test('returns the current user', () => {
    expect(ctx.res.status).toBe(200);
    expect(body.id).toBeDefined();
  });
}
```

<Callout type="warn" title="Active test context required">
  `describe`, `test`, and `expect` are active only inside the
  asynchronous `onTest` context. Calls from `onRequest`, `onResponse`,
  module initialization, or a completed hook are no-ops.
</Callout>

## Lifecycle

Test mode follows the complete request lifecycle:

```d
workspace/group/entity onRequest
  → mocked or real response
  → workspace/group/entity onResponse
  → workspace/group/entity onTest
```

`ctx` contains the final standard Web `Request`, the read-only Web
`Response`, `isMockResponse`, workspace APIs, execution metadata, and
the execution's `AbortSignal`. Use `clone()` before reading a body
more than once.

## Imports and globals

The testing functions are available as globals and through the
generated virtual module:

```ts
import { describe, expect, test } from 'yasumu:test';
```

Both forms share the same isolated test context.

## Grouping tests

Suites may be nested. Registration and result ordering are
deterministic:

```ts
export function onTest(ctx: TestHookContext) {
  describe('users', () => {
    describe('response metadata', () => {
      test('is JSON', () => {
        expect(ctx.res.headers.get('content-type')).toContain(
          'application/json',
        );
      });
    });
  });
}
```

Suite paths are included in serializable results.

## Async tests

Test callbacks may return promises:

```ts
export function onTest(ctx: TestHookContext) {
  test('contains a user', async () => {
    const body = await ctx.res.clone().json();
    expect(body.user.email).toContain('@');
  });
}
```

Thrown errors and rejected promises become failed test results. Test
execution honors the hook timeout and cancellation signal.

## Test controls

Each callback receives a control object:

```ts
export function onTest(ctx: TestHookContext) {
  test('production-only assertion', (testContext) => {
    if (ctx.workspace.env.name !== 'Production') testContext.skip();
    expect(ctx.res.ok).toBe(true);
  });

  test('explicit result', (testContext) => {
    if (ctx.res.status === 204) testContext.succeed();
    if (!ctx.res.ok) testContext.fail(`Unexpected ${ctx.res.status}`);
  });
}
```

`skip()`, `fail(message?)`, and `succeed()` end the current test with
the corresponding result.

## Common matchers

Yasumu's `expect` API follows familiar Jest/Vitest conventions,
including:

- equality: `toBe`, `toEqual`, `toStrictEqual`, `toMatchObject`
- presence: `toBeDefined`, `toBeUndefined`, `toBeNull`
- truth: `toBeTruthy`, `toBeFalsy`
- collections: `toContain`, `toContainEqual`, `toHaveLength`
- numbers: `toBeGreaterThan`, `toBeLessThan`, `toBeCloseTo`
- objects: `toHaveProperty`, `toBeInstanceOf`
- errors: `toThrow`
- modifiers: `.not`, `.resolves`, `.rejects`

```ts
export async function onTest(ctx: TestHookContext) {
  const data = await ctx.res.clone().json();

  test('response contract', () => {
    expect(ctx.res.ok).toBe(true);
    expect(data).toMatchObject({ active: true });
    expect(data.roles).toContain('user');
  });
}
```

## Mocks and nested requests

Tests run for responses returned by `onRequest` exactly as they do for
network responses. `ctx.isMockResponse` lets a test assert which path
ran.

Nested REST and GraphQL runs can enable tests with `runTests: true`.
Their test results, logs, diagnostics, and child execution ID are
returned in the nested summary and remain observable through execution
events.

## Isolation

Each runtime session owns its test registration. Concurrent runs,
separate workers, and separate workspaces do not share suites or
assertions. The Node adapter uses asynchronous context internally;
Tanxium implements the same contract without exposing runtime-specific
state to scripts.

## CLI usage

```bash
# One entity
yasumu test get-current-user

# Every REST and GraphQL entity
yasumu test --environment CI

# Machine-readable CI result
yasumu test --environment CI --json
```

The CLI exits with `1` when an assertion or execution fails, `2` for
invalid workspace or input, and `130` when cancelled.

## Complete example

```ts
import { describe, expect, test } from 'yasumu:test';

export function onRequest(ctx: RequestHookContext) {
  const headers = new Headers(ctx.req.headers);
  headers.set('accept', 'application/json');
  ctx.setRequest(new Request(ctx.req, { headers }));
}

export async function onResponse(ctx: ResponseHookContext) {
  console.log('status', ctx.res.status);
}

export async function onTest(ctx: TestHookContext) {
  const body = await ctx.res.clone().json();

  describe('GET user', () => {
    test('succeeds', () => expect(ctx.res.status).toBe(200));
    test('has an ID', () => expect(body.id).toBeDefined());
  });
}
```
