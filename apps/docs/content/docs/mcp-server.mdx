# MCP Server

<Callout type="info" title="Beta Feature">
  The GraphQL module is currently in active development. Some features
  may change as we iterate. Your feedback is welcome!
</Callout>

Yasumu includes a local Model Context Protocol server that lets an MCP
client inspect, edit, synchronize, and execute workspace requests. It
runs inside the Tanxium runtime while the desktop app is open, so MCP
uses the same RPC services, headless execution lifecycle, and
persistence layer as the application.

The server is designed for local automation. A compatible agent can
create or update a saved request, execute REST and GraphQL entities,
inspect canonical execution results, and synchronize workspace files
without switching to the desktop UI.

## When to Use It

Use the MCP server when you want an AI agent or another local tool to:

- discover workspaces and saved REST or GraphQL requests
- create, update, or delete saved requests
- add JavaScript lifecycle scripts to REST requests
- run REST and GraphQL requests through Yasumu's canonical lifecycle
- execute request tests and inspect structured test results
- cancel a running execution
- synchronize the workspace to disk after edits

The MCP server is not a remote API. It binds to `127.0.0.1` and should
be treated as a local desktop automation endpoint.

## Transport

The server uses Streamable HTTP JSON-RPC at the root path:

```http
POST http://127.0.0.1:<port>/
Content-Type: application/json
```

The health endpoint is available at the same root path:

```http
GET http://127.0.0.1:<port>/
```

Some MCP clients open an SSE stream before or after the Streamable
HTTP connection. Yasumu supports that compatibility path:

```http
GET http://127.0.0.1:<port>/
Accept: text/event-stream
```

The active port is assigned automatically when Yasumu starts. You can
find it from the MCP status item in the Yasumu status bar. Opening the
status item navigates to the health endpoint.

## Client Setup

Most clients only need the server URL:

```text
http://127.0.0.1:<port>/
```

If your client asks for a transport, choose Streamable HTTP. If the
client also supports SSE fallback, leave it enabled. Yasumu returns
JSON for normal HTTP requests and `text/event-stream` only when the
client explicitly asks for SSE through the `Accept` header.

## Protocol Calls

Initialize the server:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-03-26",
    "capabilities": {},
    "clientInfo": {
      "name": "example-client",
      "version": "1.0.0"
    }
  }
}
```

List tools:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list",
  "params": {}
}
```

Call a tool:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "workspaces_list",
    "arguments": {
      "take": 20
    }
  }
}
```

Notifications are accepted and return HTTP `202` with no JSON-RPC
body. Yasumu also handles the MCP `notifications/cancelled`
notification for active tool calls.

## Workspace Resolution

Tools that operate on workspace entities accept `workspaceId`. The
value can be any of these:

- workspace id
- workspace name
- full workspace path
- workspace folder name

If `workspaceId` is omitted, Yasumu uses the active workspace. Passing
the workspace explicitly is recommended for agents because it avoids
accidentally editing whichever workspace is active in the UI.

For example, all of these can resolve the same workspace if they match
an existing workspace record:

```json
{ "workspaceId": "u3slh2..." }
```

```json
{ "workspaceId": "test-workspace" }
```

```json
{ "workspaceId": "D:\\work\\neplex\\yasumu\\test-workspace" }
```

## Tools

| Tool                          | Description                                              |
| :---------------------------- | :------------------------------------------------------- |
| `workspaces_list`             | List recent workspaces                                   |
| `workspaces_active`           | Return the active workspace id                           |
| `workspaces_get`              | Get a workspace by id                                    |
| `rest_list`                   | List REST requests in a workspace                        |
| `rest_get`                    | Get a REST request by id                                 |
| `rest_create`                 | Create a REST request                                    |
| `rest_update`                 | Update supported REST request fields                     |
| `rest_update_script`          | Replace the saved JavaScript lifecycle script            |
| `rest_execute`                | Execute REST through the canonical headless lifecycle    |
| `rest_delete`                 | Delete a REST request by id                              |
| `graphql_list`                | List GraphQL requests in a workspace                     |
| `graphql_get`                 | Get a GraphQL request by id                              |
| `graphql_create`              | Create a GraphQL request                                 |
| `graphql_update`              | Update a GraphQL request by id                           |
| `graphql_execute`             | Execute GraphQL through the canonical headless lifecycle |
| `graphql_delete`              | Delete a GraphQL request by id                           |
| `synchronization_synchronize` | Synchronize a workspace with its `yasumu` folder         |

## Canonical Request Execution

`rest_execute` and `graphql_execute` both call the same
`execution.execute` RPC command used by the desktop application. MCP
does not implement a separate HTTP, interpolation, script, mock, or
test lifecycle. The canonical headless service owns all of those
behaviors.

The tools return the `ExecutionResult` from that command without
reshaping it. This means REST and GraphQL expose the same statuses,
request and response snapshots, mocks, tests, logs, diagnostics,
nested executions, and serialized errors as other Yasumu hosts.

### Mode Defaults

- `rest_execute` defaults to `mode: "test"` for compatibility with the
  original MCP tool. It runs request and response hooks and also runs
  test hooks.
- `graphql_execute` defaults to `mode: "run"`. It runs the request and
  response lifecycle but skips test hooks.
- Pass `mode: "run"` or `mode: "test"` explicitly whenever client
  behavior should not depend on a default.

### Supported Execution Inputs

Both execution tools accept these arguments:

| Argument               | Type              | Behavior                                                     |
| :--------------------- | :---------------- | :----------------------------------------------------------- |
| `workspaceId`          | string            | Workspace id, name, path, or folder name; defaults to active |
| `id`                   | string            | Required saved REST or GraphQL entity id                     |
| `executionId`          | string            | Optional stable id supplied by the caller                    |
| `environmentId`        | string            | Selects an environment instead of the active environment     |
| `mode`                 | `"run" \| "test"` | Controls whether test hooks run                              |
| `pathParameters`       | object            | JSON path parameter values by name                           |
| `variables`            | object            | Temporary JSON interpolation values                          |
| `secrets`              | object            | Temporary string secrets; result snapshots remain redacted   |
| `timeoutMs`            | number            | Non-negative overall execution timeout in milliseconds       |
| `scriptTimeoutMs`      | number            | Non-negative timeout for each script hook                    |
| `includeResponseBody`  | boolean           | Controls response body inclusion in the snapshot             |
| `maxRequestBodyBytes`  | number            | Maximum request body bytes captured                          |
| `maxResponseBodyBytes` | number            | Maximum response body bytes captured                         |
| `followRedirects`      | boolean           | Controls transport redirect handling                         |

`variables` and `pathParameters` must contain JSON-serializable
values. Every `secrets` value must be a string.

### Execute REST

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "method": "tools/call",
  "params": {
    "name": "rest_execute",
    "arguments": {
      "workspaceId": "test-workspace",
      "id": "q12allkockl4taxtxuhul9rb",
      "executionId": "agent-run-42",
      "mode": "test",
      "environmentId": "staging",
      "pathParameters": {
        "name": "pikachu"
      },
      "variables": {
        "region": "eu-west-1",
        "page": 2
      },
      "secrets": {
        "token": "temporary-token"
      },
      "timeoutMs": 60000,
      "includeResponseBody": true
    }
  }
}
```

For compatibility, REST also accepts `pathParams` as an alias for
`pathParameters`. Legacy values may use an enabled wrapper:

```json
{
  "pathParams": {
    "name": {
      "value": "pikachu",
      "enabled": true
    },
    "ignored": {
      "value": "not-used",
      "enabled": false
    }
  }
}
```

Disabled legacy entries are omitted before calling
`execution.execute`.

### Execute GraphQL

```json
{
  "jsonrpc": "2.0",
  "id": 5,
  "method": "tools/call",
  "params": {
    "name": "graphql_execute",
    "arguments": {
      "workspaceId": "test-workspace",
      "id": "graphql-request-id",
      "mode": "run",
      "variables": {
        "viewerId": 42
      }
    }
  }
}
```

GraphQL request construction, interpolation, hooks, mocks, tests, and
errors follow the same headless execution contract as REST.

## REST Scripts and Legacy Inputs

Use `rest_update_script` to replace the saved JavaScript lifecycle
script:

```json
{
  "jsonrpc": "2.0",
  "id": 6,
  "method": "tools/call",
  "params": {
    "name": "rest_update_script",
    "arguments": {
      "workspaceId": "test-workspace",
      "id": "q12allkockl4taxtxuhul9rb",
      "script": "export function onRequest(ctx) { return Response.json({ source: 'mock' }); }"
    }
  }
}
```

`rest_execute` retains its original `script` and `persistScript`
fields. A supplied script must be persisted before canonical
execution:

```json
{
  "name": "rest_execute",
  "arguments": {
    "workspaceId": "test-workspace",
    "id": "q12allkockl4taxtxuhul9rb",
    "script": "export function onTest() { test('status is ok', () => expect(response.status).toBe(200)); }",
    "persistScript": true,
    "mode": "test"
  }
}
```

Yasumu first updates the saved REST entity and then calls
`execution.execute`. It does not create a temporary execution-only
script because that would make MCP behave differently from the other
hosts.

<Callout type="warning" title="Legacy execution overrides">
  Non-empty `queryParams` overrides are rejected with
  `MCP_INVALID_TOOL_INPUT`. Update the saved request query parameters
  through `rest_update` before execution. A `script` without
  `persistScript: true` is also rejected with
  `MCP_INVALID_TOOL_INPUT`; use `rest_update_script` or persist the
  supplied script. `pathParams` remains supported as an alias for
  `pathParameters`.
</Callout>

## Execution Results

The MCP `tools/call` result contains one text content item. Its text
is the pretty-printed JSON serialization of the exact canonical
`ExecutionResult`:

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\n  \"executionId\": \"agent-run-42\",\n  ...\n}"
      }
    ]
  }
}
```

After parsing `content[0].text`, the result has this shared shape:

```json
{
  "executionId": "agent-run-42",
  "rootExecutionId": "agent-run-42",
  "entityId": "q12allkockl4taxtxuhul9rb",
  "entityKind": "rest",
  "request": {
    "url": "https://pokeapi.co/api/v2/pokemon/pikachu",
    "method": "GET",
    "headers": [["accept", "application/json"]],
    "body": {
      "kind": "empty",
      "size": 0,
      "truncated": false
    }
  },
  "response": {
    "status": 200,
    "statusText": "OK",
    "headers": [["content-type", "application/json"]],
    "body": {
      "kind": "json",
      "value": {
        "name": "pikachu"
      },
      "size": 18,
      "truncated": false,
      "contentType": "application/json"
    }
  },
  "isMockResponse": true,
  "status": "completed",
  "startedAt": 1750000000000,
  "completedAt": 1750000000148,
  "durationMs": 148,
  "tests": [
    {
      "id": "status-test",
      "suite": ["Pokemon"],
      "test": "returns 200",
      "result": "pass",
      "error": null,
      "duration": 2
    }
  ],
  "logs": [],
  "diagnostics": [],
  "nestedExecutions": []
}
```

`request` and `response` use bounded serialized body snapshots instead
of the old MCP-only `textBody` representation. `response` may be
absent when execution fails before a response exists. `isMockResponse`
is `true` when a request hook supplies the response. Nested executions
and structured errors are returned without MCP-specific conversion.

### Failed Executions and Tool Errors

An execution failure or cancellation is still a valid canonical
result. Inspect `status` and `error`:

```json
{
  "status": "failed",
  "error": {
    "name": "YasumuError",
    "code": "REQUEST_FAILED",
    "message": "Connection refused",
    "details": {
      "retryable": true
    }
  }
}
```

Input validation and RPC invocation errors are MCP tool errors. Yasumu
returns JSON-RPC success with `isError: true` and a structured text
payload:

```json
{
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\n  \"error\": {\n    \"code\": \"MCP_INVALID_TOOL_INPUT\",\n    \"message\": \"...\",\n    \"details\": { \"field\": \"queryParams\" }\n  }\n}"
      }
    ],
    "isError": true
  }
}
```

## Cancellation

Yasumu assigns an execution id before calling `execution.execute`.
Pass `executionId` when your client needs a stable value for logs or
result correlation; otherwise Yasumu generates one.

There are two cancellation paths:

- aborting the Streamable HTTP request forwards cancellation to
  `execution.cancel`
- sending `notifications/cancelled` for the active JSON-RPC request id
  forwards the notification reason to `execution.cancel`

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": {
    "requestId": 4,
    "reason": "No longer needed"
  }
}
```

`requestId` is the JSON-RPC id of the active `tools/call`, not the
Yasumu execution id. The notification receives HTTP `202`. A
successfully cancelled execution resolves with `status: "cancelled"`
and its structured execution error.

## Synchronization

After an agent edits workspace entities, call
`synchronization_synchronize` to write the latest workspace state to
the workspace `yasumu` folder:

```json
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "synchronization_synchronize",
    "arguments": {
      "workspaceId": "test-workspace"
    }
  }
}
```

This keeps the desktop database state and the serialized workspace
files aligned for collaboration and version control.

## Security and Privacy

The MCP server is local-only and does not bind to external network
interfaces. It does not expose shell execution or arbitrary filesystem
tools. All workspace operations go through Yasumu's Tanxium RPC
services.

Execution secrets are passed to the canonical headless service and are
redacted from its returned snapshots, logs, diagnostics, and errors.
Agents should still avoid intentionally writing secrets into remote
request bodies or script-controlled output.

## Troubleshooting

### Method Not Found During Connect

If a client reports `Method not found` during initialization, verify
that it is posting JSON-RPC to the root endpoint:

```text
http://127.0.0.1:<port>/
```

Older or strict clients may also try SSE fallback. Yasumu supports SSE
only when the request includes `Accept: text/event-stream`.

### Invalid Content Type for SSE

This usually means the client requested the health endpoint as normal
HTTP instead of an SSE stream. Configure the client for Streamable
HTTP and keep the URL at the root path.

### Workspace Not Found

Use `workspaces_list` first and pass the exact workspace id returned
by Yasumu. Workspace names and folder names are accepted for
convenience, but ids are the most predictable option.

### Request Execution Fails

Parse `content[0].text`, then inspect the canonical `status`, `error`,
and `diagnostics` fields. Common causes include:

- the saved request has no valid URL or GraphQL operation
- a required interpolation or path parameter is missing
- the remote server is unavailable
- the execution or script hook timed out
- a lifecycle script produced an invalid request

For a tool-level error, check `result.isError` and parse the
structured error in `content[0].text`.

### Script Runs but Tests Are Missing

Pass `mode: "test"` and ensure the saved script exports an `onTest`
handler. `graphql_execute` defaults to run mode, so its test hooks do
not run unless test mode is explicit.
