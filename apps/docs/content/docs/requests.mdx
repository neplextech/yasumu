# Requests

REST entities are stored in YSL, normalized into the shared workspace model,
and executed by the same headless pipeline in the desktop app and CLI.

## Creating a request

1. Open **REST** in the workspace sidebar.
2. Select **New request**.
3. Choose a method and URL.
4. Configure parameters, headers, authentication, body, and scripts.
5. Send the request or run it in test mode.

![Request Editor](/preview.png)

## YSL structure

```d
@rest

metadata {
  name: "Create user"
  method: "POST"
  id: "create-user"
  groupId: null
}

request {
  url: "{{API_URL}}/users/:userId"
  headers: [
    { key: "authorization", value: "Bearer {{API_TOKEN}}", enabled: true },
  ]
  parameters: [
    { key: "userId", value: "{{USER_ID}}", enabled: true },
  ]
  searchParameters: []
  body: {
    type: "json"
    content: "{ \"name\": \"{{DISPLAY_NAME}}\" }"
  }
}

dependencies []
script null
test null
```

Entity IDs remain stable across the filesystem, SQLite import, CLI, scripts,
and execution history. The loader rejects duplicate IDs and invalid group or
dependency references with source-located diagnostics.

## Supported request data

Yasumu supports standard HTTP methods and these body types:

| Body type | Behavior |
| :-------- | :------- |
| JSON | Preserves typed whole-value interpolation |
| Text | Sends text with deterministic encoding |
| Binary | Resolves a runtime-independent file reference |
| Form data | Supports repeated text fields and multiple files |
| URL encoded | Builds `application/x-www-form-urlencoded` pairs |

Enabled headers, path parameters, and search parameters are interpolated at
execution time. Disabled rows are ignored.

## Variable interpolation

Use `{{NAME}}` in URLs, keys, headers, parameters, bodies, GraphQL variables,
and eligible file paths:

```d
url: "{{API_URL}}/teams/:teamId"
```

When a JSON or GraphQL value consists entirely of one reference, booleans,
numbers, `null`, arrays, and objects keep their type. Embedded references are
rendered as strings. Recursive values have a depth limit and cycles or missing
values produce structured errors.

See [Environments](/docs/environments) for precedence and secret handling.

## Files and form data

File values use a serializable reference whose source is one of:

- a workspace-relative path
- a desktop host handle
- inline bytes selected by the renderer

The desktop resolves GUI selections in the backend. The CLI resolves YSL paths
relative to the workspace and blocks traversal or symlink escape. Scripts can
inspect metadata or intentionally open workspace files through
`ctx.workspace.files`.

Multipart requests preserve repeated fields, file names, MIME types, multiple
files, and empty files. Request snapshots record metadata and bounded bodies;
they do not duplicate unbounded binary content across workers.

## Script lifecycle

Scripts use standard Web APIs:

```ts
export function onRequest(ctx: RequestHookContext) {
  const headers = new Headers(ctx.req.headers);
  headers.set('x-execution-id', ctx.execution.id);
  ctx.setRequest(new Request(ctx.req, { headers }));
}

export async function onResponse(ctx: ResponseHookContext) {
  console.log(ctx.res.status, await ctx.res.clone().text());
}

export function onTest(ctx: TestHookContext) {
  test('succeeds', () => expect(ctx.res.ok).toBe(true));
}
```

Workspace and nested group hooks cascade before the entity hook. Returning a
Web `Response` from `onRequest` skips transport but still runs `onResponse` and
`onTest`. Read [Scripting](/docs/scripting) for the complete contract.

## Execution results

The headless result is serializable and includes:

- execution, root, parent, workspace, and entity identity
- bounded request and response snapshots
- real-versus-mock status
- completion, cancellation, or failure status
- timing
- tests, logs, diagnostics, and nested executions
- a redacted structured error when execution fails

The desktop maps this result into the existing response panel and subscribes
to execution events for live phases and console output. The CLI renders the
same underlying result as human-readable or JSON output.

## Cancellation and timeouts

Cancelling from the desktop or pressing `Ctrl+C` in the CLI aborts transport,
hooks, tests, nested requests, file work, email waits, and the runtime session.
Timeouts and user cancellation remain distinct error categories.

## Dependencies and chaining

YSL dependencies are validated as references. Scripts can directly execute
another REST or GraphQL entity through `ctx.workspace.rest.execute()` or
`ctx.workspace.graphql.execute()`. Nested execution inherits the active
environment and cancellation signal, emits child events, and has a maximum
depth to prevent recursion.

## Best practices

1. Use stable IDs and descriptive names.
2. Keep endpoints and credentials in environments.
3. Store secrets outside YSL and supply them explicitly in headless runs.
4. Prefer Web `Request`/`Response` APIs in scripts.
5. Use workspace paths, not absolute paths, for portable fixtures.
6. Add `onTest` assertions so the same workspace can run in CI.
