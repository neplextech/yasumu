# SMTP Server

Each Yasumu workspace can run a local catch-all SMTP server. Messages are
stored in the workspace mailbox, appear in the desktop app in real time, and
can participate in request executions and scripts.

<Callout type="warn" title="Development use only">
  The built-in SMTP server is intended for local development and automated
  testing. Do not expose it as a production mail server.
</Callout>

## Mailbox

Open **Emails** in the workspace sidebar to:

- inspect sender, recipients, subject, text, and HTML
- filter messages
- mark mail read or unread
- delete individual messages or clear the mailbox
- edit the workspace's incoming-email script
- start, stop, and configure the SMTP server

Messages received while an API execution is running are available to that
execution's email API even when they arrive before the script starts waiting.

## Configuration

The workspace SMTP configuration contains a stable ID, port, optional
username, optional local password, and an optional `onEmail` script. Exported
YSL does not write a usable password.

```d
@smtp

metadata {
  id: "workspace-smtp"
  port: 1025
  username: null
  password: null
}

script null
```

Common local ports are `1025` and `2525`. Ensure the selected port is not
already in use.

## Sending mail to Yasumu

### Nodemailer

```ts
import nodemailer from 'nodemailer';

const transport = nodemailer.createTransport({
  host: '127.0.0.1',
  port: 1025,
  secure: false,
});

await transport.sendMail({
  from: 'app@example.test',
  to: 'user@example.test',
  subject: 'Your verification code',
  text: 'Code: 123456',
  html: '<p>Code: <strong>123456</strong></p>',
});
```

### Command line

```bash
swaks --server 127.0.0.1:1025 \
  --from app@example.test \
  --to user@example.test \
  --header "Subject: Local test" \
  --body "Hello from the test suite"
```

## Email scripting

The SMTP script exports one optional hook. It receives the same base context
as request scripts, with a structured `ctx.email` instead of a request or
response:

```ts
export function onEmail(ctx: EmailHookContext) {
  const { email } = ctx;
  console.log('received', email.subject, 'from', email.from);
}
```

`WorkspaceEmail` contains:

| Property | Type | Description |
| :------- | :--- | :---------- |
| `id` | `string` | Stable message ID |
| `from` | `string` | Sender address |
| `to` | `string[]` | Recipient addresses |
| `cc` | `string[]` | CC addresses |
| `subject` | `string` | Subject line |
| `text` | `string` | Plain-text body |
| `html` | `string` | HTML body |
| `createdAt` | `number` | Receive timestamp in milliseconds |
| `updatedAt` | `number \| undefined` | Last-update timestamp |
| `unread` | `boolean \| undefined` | Read state |

The context also exposes `id`, `entity`, `execution`, `signal`, `cancel()`, and
the complete `ctx.workspace` API.

### Extract a verification code

```ts
export function onEmail(ctx: EmailHookContext) {
  const { email } = ctx;
  if (!email.subject.toLowerCase().includes('verification')) return;

  const code = email.text.match(/\b\d{6}\b/)?.[0];
  if (code) ctx.workspace.env.setVariable('VERIFICATION_CODE', code);
}
```

Environment changes made by scripts are execution-scoped. Persisting a value
requires an explicit host or user action; scripts never silently write secrets
to YSL or SQLite.

### Chain another entity

```ts
export async function onEmail(ctx: EmailHookContext) {
  if (!ctx.email.subject.includes('Account ready')) return;

  await ctx.workspace.rest.execute('confirm-account', {
    variables: { recipient: ctx.email.to[0] ?? '' },
  });
}
```

Nested runs inherit cancellation and emit their own child execution events.

## Waiting for mail from a request

Request scripts can wait inline through `ctx.workspace.email`:

```ts
export async function onResponse(ctx: ResponseHookContext) {
  const email = await ctx.workspace.email.awaitEmail(
    (candidate) =>
      candidate.to.includes('user@example.test') &&
      candidate.subject.includes('verification'),
    { timeoutMs: 30_000, signal: ctx.signal },
  );

  if (!email.text.includes('Code:')) throw new Error('Verification code missing');
}
```

The lifecycle-safe algorithm:

1. records the API execution start time
2. checks persisted messages received since that time
3. evaluates the predicate in the script runtime
4. waits for future candidates only when none match
5. honors timeout and cancellation
6. removes subscriptions when the wait ends

This closes the race where mail arrives after the request starts but before
`awaitEmail` is called. Multiple waits maintain independent cursors.

Use `ctx.workspace.email.list({ since, limit })` when no wait is needed.

## Diagnostics and privacy

Email timeouts, permission denial, runtime failures, and cancellation use the
same structured error model as request execution. SMTP credentials and active
environment secrets are redacted from logs, diagnostics, and history.

## Troubleshooting

- **Connection refused:** start the workspace SMTP server and verify the port.
- **Port already in use:** choose another local port and update the sender.
- **Authentication fails:** make the sender credentials match the workspace
  configuration.
- **A wait times out:** verify the recipient/subject predicate and increase
  `timeoutMs` when the sender is intentionally slow.
- **A script does not run:** ensure it exports `onEmail(ctx)`; missing hooks are
  valid no-ops.
