# Environments

Environments in Yasumu allow you to manage different configurations
for development, staging, production, and other contexts. Switch
between environments to test your APIs against different backends
without modifying individual requests.

![Environments](/environments.png)

## What Are Environments?

An environment is a collection of variables and secrets that define
context-specific values:

- **Base URLs** — Different API endpoints per environment
- **API Keys** — Different credentials per environment
- **Configuration** — Environment-specific settings

## Environment File Structure

Environments are stored in the `environment/` folder as `.ysl` files:

```d
yasumu/
└── environment/
    ├── h49ehcz1o96rglpym0qjqr3a.ysl   # Production
    ├── z85bde4ytdta7d1u0ndjfoh6.ysl   # Staging
    └── ipapmufxwgm558jh1cpfbkwe.ysl   # Nightly
```

### File Format

```d
@environment

metadata {
  id: "ipapmufxwgm558jh1cpfbkwe"
  name: "Nightly"
}

variables [
  {
    key: "API_URL"
    value: "https://echo.yasumu.local"
    enabled: true
  },
]

secrets [
  {
    key: "API_KEY"
    value: ""
    enabled: true
  },
]
```

### Sections

| Section     | Description                              |
| :---------- | :--------------------------------------- |
| `metadata`  | Environment ID and display name          |
| `variables` | Non-sensitive configuration values       |
| `secrets`   | Sensitive values (stored empty in files) |

## Variables

Variables are key-value pairs that can be used in requests:

```d
variables [
  {
    key: "API_URL"
    value: "https://api.example.com"
    enabled: true
  },
  {
    key: "API_VERSION"
    value: "v1"
    enabled: true
  },
]
```

Each variable has:

- `key` — Variable name used in requests (e.g., `{{API_URL}}`)
- `value` — The actual value
- `enabled` — Whether the variable is active

## Secrets

Secrets are sensitive values like API keys and tokens:

```d
secrets [
  {
    key: "API_KEY"
    value: ""
    enabled: true
  },
]
```

<Callout type="warn" title="Secret Storage">
  Secret values are **never written to `.ysl` files**. The `value`
  field is always stored as an empty string. Actual secret values are
  stored securely and locally on your machine, separate from version
  control.
</Callout>

This design ensures:

- **Safe to commit** — `.ysl` files can be safely committed to Git
- **No accidental leaks** — Secrets never appear in version control
- **Local override** — Each team member manages their own secret
  values

## Using Variables

Reference environment variables in your requests using double curly
braces:

```d
url: "{{API_URL}}/users?apiKey={{API_KEY}}"
```

When you send a request, the shared headless resolver builds one
effective environment for the complete hook and request lifecycle. The
desktop supplies locally stored secret values; the CLI requires
explicit secret sources.

From highest to lowest precedence:

```d
programmatic execution values
  → CLI or host overrides
  → process environment
  → selected dotenv values
  → selected environment
  → workspace defaults
```

For the CLI, process values may use `YASUMU_VAR_<NAME>` for variables
and `YASUMU_ENV_<NAME>` for defined secrets.

## Switching Environments

Switch environments using the environment selector in the toolbar.
When you switch environments, all variable references are resolved
using the new environment's values.

## Script access

Scripts read the active environment through `ctx.workspace.env`:

```ts
export function onRequest(ctx: RequestHookContext) {
  const token = ctx.workspace.env.getSecret('API_TOKEN');
  const headers = new Headers(ctx.req.headers);
  if (token) headers.set('authorization', `Bearer ${token}`);
  ctx.setRequest(new Request(ctx.req, { headers }));
}
```

`setVariable()` and `setSecret()` modify execution-scoped state for
later hooks and nested requests. They do not silently persist to YSL
or SQLite.

Secrets are centrally redacted from logs, diagnostics,
request/response snapshots, test output, errors, and execution
history.

## Typed interpolation

When a JSON body or GraphQL variable is exactly `{{VALUE}}`, the
resolved boolean, number, `null`, array, or object keeps its type.
References embedded inside a larger string become text. Missing
values, recursive cycles, and excessive expansion depth produce
structured interpolation errors instead of silently sending an
unresolved request.

## Best Practices

1. **Use clear variable names** — `API_URL` is better than `URL1`
2. **Separate secrets from variables** — Use secrets for anything
   sensitive
3. **Commit your environments** — The `.ysl` files are safe to commit
   since secrets are empty
4. **Share variable structure** — Keep the same variable keys across
   environments
5. **Enable/disable as needed** — Use the `enabled` flag to
   temporarily disable variables
